# 上线前验证方案（AI Story Studio）

本文档用于 **预发布 / 生产上线** 前的系统化验证。建议按顺序执行：**门禁 → 环境 → 数据 → 功能 → 安全 → 性能 → 发布与回滚**。

---

## 1. 验证目标与范围

| 维度 | 目标 |
|------|------|
| **可用性** | 核心路径（作品列表 → 打开编辑 → 保存 → 刷新一致）无阻塞 |
| **数据一致性** | OSS `chapters.json` / `*.txt` / `novel.json` 与界面展示一致 |
| **AI / 外部依赖** | 至少一条大模型链路可用；失败时有明确提示、不静默丢数据 |
| **安全** | 密钥不落日志、不通过未鉴权接口泄露；生产禁用或保护调试接口 |
| **可运维** | 可构建、可启动、有可执行的 OSS/索引校验命令 |

**不在此文档内展开的**：法务合规、应用商店审核、域名备案等（按实际部署平台补充）。

---

## 2. 角色与环境矩阵

| 环境 | 用途 | 建议 |
|------|------|------|
| **本地** | 开发自测 | `.env.local`，可接生产只读 OSS 副本或独立测试 Bucket |
| **预发布（Staging）** | 上线前终验 | 独立 `OSS_BUCKET`、`DATABASE_URL`，与生产数据隔离 |
| **生产（Production）** | 真实用户 | 仅平台注入环境变量，禁止把 `.env` 提交仓库 |

**上线前必须在「与生产同配置类型」的环境跑通一次完整清单**（至少 Staging）。

### 2.1 Staging：与生产「同级配置」怎么验

目标：**环境变量键名与生产完全一致**，仅值为 **Staging 专用资源**，避免「Staging 缺变量、上线后生产才爆」的问题。

| 步骤 | 操作 |
|------|------|
| 1 | 在部署平台（Vercel / 自建 Docker / 云主机等）创建 **Staging 应用**，与生产使用 **同一套变量名**（见 `.env.example`）。 |
| 2 | **OSS**：使用 **独立 Bucket** 或独立前缀策略；**禁止** Staging 直接写生产 Bucket（除非明确做只读副本且团队知情）。 |
| 3 | **DATABASE_URL**：Staging 使用 **独立** SQLite 路径或数据库文件，与生产文件分离。 |
| 4 | **AI**：可使用与生产相同供应商的 Key，或单独测试 Key；**至少验证一条链路**（见 §7）。 |
| 5 | **NODE_ENV**：构建与运行时应为 `production`（与线上一致），以便暴露相同代码分支行为。 |
| 6 | 部署 Staging 后，用下面「一键门禁」在**本地**再确认一次构建通过（与 CI 一致）。 |

**Staging 部署后的 HTTP 抽检**（将 `BASE` 换成 Staging 根 URL，需已配置 OSS）：

```bash
# 只读：OSS 元数据与抽样章节
curl -sS "$BASE/api/oss/verify" | head -c 2000

# 只读：全书索引 vs .txt（体量大会较慢）
curl -sS "$BASE/api/oss/verify-chapter-files" | head -c 4000
```

**手工 P0（Staging 上浏览器）**：打开首页 → 进入一本书 → 章节目录滚到底 → 编辑保存 → 刷新仍一致 → 做一次短 AI 动作（若启用）。

## 3. 上线门禁（自动化，必须通过）

在**将要部署的同一 Node 版本**下执行：

```bash
# 依赖安装（与 CI 一致）
npm ci

# 等价于：lint + build（与 GitHub Actions 一致）
npm run ci:release
```

或分步：

```bash
npm run lint
npm run build
```

**期望结果**：

- `lint` 无 error（warning 建议清零或记录在案）。
- `build` 成功退出码 0。

**可选加强**（若团队接受维护成本）：

- 在 CI 中增加 `npx tsc --noEmit`（当前仓库若有个别历史 TS 报错，需先修复或收窄检查范围）。
- 对关键 `src/lib/*.ts` 增加单元测试（字数统计、章节排序等）。

## 4. 环境变量核对清单

对照仓库根目录 `.env.example`，在**部署平台**逐项确认（勿把真实值写入文档或截图）：

| 变量 | 用途 | 验证方式 |
|------|------|----------|
| `DATABASE_URL` | SQLite 路径（修订历史等） | 启动后创建/读取章节修订不报错 |
| `OSS_REGION` / `OSS_ACCESS_KEY_ID` / `OSS_ACCESS_KEY_SECRET` / `OSS_BUCKET` | 作品主存储 | 见第 5、6 节 |
| `ZAI_API_KEY` 或 `ALIYUN_AI_API_KEY` 或 `ZHIPU_AI_API_KEY` | 大模型（优先级见示例） | 见第 7.2 节 |
| 各 `*_BASE_URL` / `*_MODEL` | 模型路由 | `/api/debug` 或一次最小 AI 请求 |

**常见故障**：

- Bucket 与 Region 不匹配 → OSS 403/签名错误。
- 生产未注入某 AI Key → 功能降级或 503，需在界面有提示。

## 5. OSS 与数据完整性验证（强烈建议）

### 5.1 只读 HTTP 校验（服务已启动时）

| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/oss/verify` | GET | 抽样校验 `listOSSNovels`、元数据、`chapters.json` 解析、部分章节读取 |
| `/api/oss/verify-chapter-files` | GET | 无 query：全书；`?novelId=`：单本。对照索引与 `chapters/*.txt` |
| `/api/oss/test` | 按实现 | 连通性探测（若存在） |

**通过标准（建议）**：

- `verify-chapter-files` 中关心书目 `report.fullyConsistent === true`，或至少无 `duplicateIndexIds`、无 `missingTxtForIndexId`、无 `orphanTxtIds`（若业务允许孤儿，需有处理流程）。

### 5.2 本地脚本（不依赖 Next 进程）

```bash
npm run verify:oss-chapters
# 或单本：
node scripts/oss-verify-chapter-index-vs-txt.js <novelId>
```

### 5.3 索引修复类操作（慎用，在维护窗口执行）

| 能力 | 说明 |
|------|------|
| `POST /api/oss/reconcile-word-counts` | 按 `.txt` 修正索引 `wordCount` 与 `novel.json` |
| `POST /api/oss/rebuild-chapter-index` | 按 `chapters/*.txt` **全量重建** `chapters.json`（需 `confirm: true`） |
| `POST /api/chapters/repair-sequence` | 仅重排章号/顺序，不扫盘 |

**验证**：在**测试 Bucket** 上对 1 本书执行 dry-run / 单本验证后再上生产。

## 6. 核心用户路径（手工 / E2E）

以下为 **P0**，每条建议在桌面端 + 至少一种移动端宽度各走一遍。

### 6.1 作品与章节

| ID | 步骤 | 期望 |
|----|------|------|
| P0-1 | 打开首页，加载作品列表 | 无白屏；字数、章数与 OSS 大致一致 |
| P0-2 | 进入一本书编辑页 | 章节目录可滚动，**最后一章可见**（侧栏布局已修） |
| P0-3 | 切换多章 | 标题、正文与当前章一致 |
| P0-4 | 编辑正文，失焦或等待自动保存 | 刷新页面后内容仍在 |
| P0-5 | 新建章节 | 列表出现新章，OSS 存在对应 `.txt` 与索引项 |
| P0-6 | 删除章节（若有） | 索引与文件符合产品预期（是否删 OSS 以实际 API 为准） |
| P0-7 | 导出（若使用） | 文件可下载且内容完整 |

### 6.2 章节检测与修复（智能生成用户）

| ID | 步骤 | 期望 |
|----|------|------|
| P1-1 | 章节检测 → 重复/缺失报告 | 与预期数据一致 |
| P1-2 | 一键修复章号与顺序 | 成功后章号连续、无重复 order 报错 |
| P1-3 | 从 OSS 正文重建索引 | 仅在意图修复索引时使用；执行后 `verify-chapter-files` 通过 |

### 6.3 作品档案 / 大纲 / 设置

| ID | 步骤 | 期望 |
|----|------|------|
| P1-4 | Story Bible 读写 | 保存后再进入数据仍在 |
| P1-5 | 大纲相关接口 | 不阻塞写作主路径 |

## 7. AI 与流式能力

### 7.1 最小联通性

任选一种：

- 在 UI 执行一次「续写 / 生成标题」等短请求；或  
- 调用 `POST /api/ai/action`（带合法 body）。

**期望**：HTTP 200，返回结构符合前端解析；超时时间可接受（记录实际 P95）。

### 7.2 长流程 / 流式（P1）

| 能力 | 验证要点 |
|------|----------|
| 智能生成 / `stream-generate` | 中断网络或刷新时，已写入 OSS 的章节不丢失；无并发写坏 `chapters.json`（已做串行写） |
| 补全缺失章 | 边界：已有重复章时行为与提示符合设计 |

### 7.3 失败与降级

- 错误 API Key：应返回明确错误，**不**覆盖已有章节正文。
- 限流 / 429：界面有重试或提示（按实现检查）。

## 9. 安全与合规（生产必查）

### 9.1 接口暴露

| 项目 | 风险 | 建议 |
|------|------|------|
| **`GET /api/debug`** | 返回 OSS/AI **配置摘要及部分 Key 前缀**，易被扫描利用 | **生产禁用**（删除路由或 `NODE_ENV==='production'` 返回 404），或仅内网 + `Authorization` 头密钥 |
| **OSS 维护类 API** | `rebuild` / `clean-orphan` / `normalize` 等可改数据 | 无用户鉴权时，勿将站点暴露公网；或加网关鉴权 / IP 白名单 |
| **CORS** | 若前后端分离 | 按域名白名单配置 |

### 9.2 密钥与日志

- 部署平台环境变量**加密存储**；禁止在客户端打包任何 `SECRET`。
- 服务器日志过滤 `Authorization`、完整 OSS 响应体中的敏感字段。

### 9.3 依赖与构建

- 定期 `npm audit`（或等价），高危漏洞在上线前评估或修复。
- 使用固定版本锁文件（`package-lock.json`）构建。

## 10. 性能与稳定性（抽样）

| 项 | 建议方法 | 备注 |
|----|----------|------|
| 大书目录 | 300+ 章的书打开编辑页 | 首屏时间、目录滚动是否卡顿 |
| 冷启动 | 生产 `npm run start` 或容器启动 | 记录启动到可访问耗时 |
| 并发写 | 两标签页同时改**不同**章节 | 两章均保存成功；同章仍以最后保存为准（预期） |

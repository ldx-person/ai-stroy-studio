# AI Story Studio × Sudowrite 对齐改造计划

> 目标：把 AI Story Studio 的产品能力对齐 Sudowrite 的「动作化写作 + 多版本试错 + 描写增强 + 一致性注入」核心体验，同时保留中文网文工作流优势（大纲 + 每章开头、不中断作者掌控）。  
> 范围：产品信息架构、前端交互、后端 API 形态、数据结构与一致性机制、验收指标。  
> 术语：下文把 Sudowrite 的能力抽象为 **AI Actions（写作动作）**。

---

## 0. 现状盘点（你项目已有能力）

基于 `PRODUCT_FEATURES.md` 与当前实现，AI Story Studio 已具备：

- **内容组织**：多本小说、章节目录、章节编辑、自动保存、字数统计。
- **AI 能力**：
  - 续写（`/api/ai/continue`）
  - 章节标题（`/api/ai/title`）
  - 润色/精简/扩写（`/api/ai/refine`）
  - 智能生成（故事结构 + 章节大纲 + 每章约 100 字开头，且不覆盖既有章节）（`/api/ai/smart-generate`/`stream-generate`）
- **一致性雏形**：智能生成过程中维护 summaries/characters（后端逻辑层面）。
- **输出消费**：结果可预览并采用；TTS 朗读。
- **存储**：本地 DB + 可选 OSS 同步。

你已经在“**每章只生成开头、作者继续写**”这一点上，比很多一键全文生成产品更符合网文现实工作流。

## 1. 对齐矩阵：Sudowrite 核心体验 vs 当前差距

| 体验维度 | Sudowrite 典型做法 | 你项目现状 | 主要差距 | 机会点 |
|---|---|---|---|---|
| **动作化写作（Action-first）** | “Continue / Rewrite / Describe / Brainstorm…”一键执行 | 已有按钮（续写/标题/润色/精简/扩写） | 动作定义不够系统：缺少“目标明确 + 输入范围 + 输出长度”标准化 | 统一成 Action 系统，扩展动作库 |
| **多版本试错（Variants）** | 同一动作一次给 3–5 个候选 | 单结果 | 缺少变体与对比采纳 | 引入 variants、对比、局部采纳 |
| **就地编辑（In-place adopt）** | 生成结果紧贴编辑器，低摩擦替换/插入 | 有预览卡片与采用 | 缺少“插入位置（光标/选区）”“对比视图” | 支持选区、插入/替换、diff |
| **描写增强（Describe）** | 感官维度扩写、氛围增强是强项 | 目前只有“扩写”泛化动作 | 动作粒度偏粗 | Describe：按感官/氛围/动作分型 |
| **一致性（长篇防遗忘）** | 借助角色卡/设定库/时间线（工具内或用户维护） | 有 OSS 结构，但未形成“作品档案”产品面板 | 缺少可视化设定管理与自动注入 | 作品档案 + 自动注入生成提示 |
| **结构化创作（大纲/场景卡）** | Outline/Scene card | 已有大纲生成与智能开头 | 缺少可编辑/可版本化的大纲 UI | 大纲树、章节卡、批量操作 |
| **成本与反馈** | 进度/失败原因/配额提示清晰 | SSE 进度已做，计费/失败提示可强化 | 缺少“请求成本/余额不足”的产品提示 | 错误码映射、配额提醒、降级策略 |

## 2. 产品信息架构改造：把“功能”变成“动作”

### 2.1 新的 AI 助手模块结构

将 `AI助手` Tab 从“按钮堆”升级为 **Action 面板**：

- **动作分类**（左侧或顶部二级导航）
  - **推进**：续写下一段、写本章开头、写下一幕（Scene）
  - **改写**：润色、精简、扩写、改成更紧凑、更口语、更古风、更悬疑
  - **描写**：环境、动作、情绪、感官（视觉/听觉/触觉/嗅觉/味觉）
  - **对话**：生成对话、让对话更有张力、补充潜台词
  - **构思**：剧情点子、冲突升级、反转建议、角色动机

### 2.2 统一的动作输入模型（非常关键）

每个动作都明确三件事：

1. **输入范围**：选区 / 光标所在段 / 当前章节 / 最近 N 段 / 最近 N 章摘要  
2. **输出形式**：插入 / 替换 / 追加到建议区  
3. **输出长度**：一句 / 100 字 / 300 字 / 800 字（你已明确智能开头约 100 字）

这能显著减少“跑偏”和“用户不知道怎么用”的问题。

## 3. 前端交互改造（对齐 Sudowrite 的使用体验）

### 3.1 编辑器增强（P0）

- **选区能力**：用户选中文字后，AI 动作默认作用于选区（改写/描写/对话最需要）。
- **插入点**：支持在光标处插入（续写/补段落）。
- **结果区**：显示 1–5 个候选卡片，每张卡片支持：
  - 替换选区 / 插入光标 / 复制
  - “再生成 1 版”（快速变体）

### 3.2 多版本对比（P1）

- **Variants**：同一动作可选 1/3/5 个候选。
- **差异对比**：提供简易 diff（可先做“原文 vs 候选”对比视图）。
- **局部采纳**：允许用户在候选中选一句/一段插入（降低“全替换”的心理阻力）。

### 3.3 动作模板与引导（P1）

- 新用户引导：给 3 个“示例动作”（续写、描写增强、改写成更紧凑）。
- 动作说明：每个动作下显示“它会做什么 + 建议输入范围 + 输出长度”。

## 4. 后端改造：AI Action API + 多版本机制

### 4.1 建议新增统一入口：`POST /api/ai/action`

将离散接口（continue/title/refine/…）逐步收敛到：

```json
{
  "novelId": "…",
  "chapterId": "…",
  "action": "continue|rewrite|describe|dialogue|brainstorm|title|opening",
  "input": {
    "scope": "selection|cursor|chapter",
    "text": "…",
    "selection": { "start": 0, "end": 120 } 
  },
  "options": {
    "variants": 3,
    "length": "100w|300w|800w",
    "style": "紧凑|古风|轻松|悬疑|…",
    "constraints": { "pov": "第一人称", "tense": "过去时" }
  }
}
```

返回：

```json
{ "success": true, "candidates": [{ "text": "..." }, { "text": "..." }] }
```

优点：前端可以用同一套 UI 渲染候选、做对比、做采纳。

### 4.2 逐步迁移策略

- P0：保留现有接口，同时新增 `/api/ai/action`，先覆盖 3 个动作：`continue`、`rewrite(polish/shorten/expand)`、`opening(100字开头)`。
- P1：把 `title`、`describe`、`dialogue` 迁入统一入口。
- P2：把“智能生成大纲 + 每章开头”接入 action 系统（批处理）。

## 5. 一致性注入（长篇防遗忘）：作品档案（Story Bible）

### 5.1 数据结构（建议落在 OSS + DB）

新增 `story_bible.json`（或拆分文件）：

- `characters[]`：角色卡（性格、动机、口癖、关系、禁忌）
- `worldRules`：世界观规则（体系、势力、地理、设定约束）
- `timeline[]`：关键事件时间线
- `styleGuide`：文风规则（叙述视角、常用句式、禁用表达）

### 5.2 自动注入策略（P1）

在每次 Action 调用时，后端自动拼接上下文：

- 最近 N 章摘要（你已有 summaries 逻辑，可落库）
- 主要角色卡（按最近章节出现的角色筛选）
- 世界观约束（常驻）

并在 system prompt 中要求：
- 不新增未声明设定
- 不改变角色基本性格与关系

## 6. 迭代里程碑（P0/P1/P2）与验收指标

### P0（1–2 周）：动作化与可控粒度

- 编辑器支持选区与插入点（最小可用）
- AI 动作面板重排：继续/润色/精简/扩写/写开头
- 每个动作支持输出长度（100/300）
- **验收指标**：
  - 用户从“选区→改写→替换”完成闭环 ≤ 10 秒（不含模型时间）
  - 续写跑偏率（主观）下降：用户平均 1–2 次内选到可用候选

### P1（2–4 周）：多版本 + 描写增强 + 一致性注入

- `/api/ai/action` 支持 variants=3
- Describe 动作（按感官/氛围）上线
- 作品档案（角色卡/世界观）可编辑 + 自动注入
- **验收指标**：
  - 同一段改写：3 版本中至少 1 版可直接采用的比例提升
  - 长篇一致性：连续 10 次生成中设定冲突显著减少（主观/抽样）

### P2（4–8 周）：结构化创作与批处理

- 大纲树/章节卡可编辑与版本化
- 批量生成“每章开头”（基于大纲批处理），并与现有章节目录对齐追加
- 简易 diff 与局部采纳体验强化

---

## 7. 与当前“智能章节生成”的融合方式（不覆盖 + 开头优先）

你已经实现的策略非常正确，建议保持并产品化：

- **默认只生成开头**：写作掌控权在作者手里。
- **只追加缺失章节**：生成前读取章节目录，避免覆盖。
- **将其包装成动作**：`Generate outlines`（生成大纲）+ `Generate openings`（批量生成开头）。

---

## 8. 下一步落地建议（最短路径）

如果你希望最快看到 Sudowrite 风味的提升，建议按顺序做：

1. **选区改写**（Rewrite）+ **3 版本候选**（Variants=3）
2. **Describe：情绪/环境** 两个动作（最常用）
3. **作品档案：角色卡**（只做 3–5 个关键字段），并自动注入
4. 最后再做大纲树/章节卡版本化
